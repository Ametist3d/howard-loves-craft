OUTPUT FORMAT — MANDATORY

Return exactly this structure and nothing else:

<SYSTEM_RESPONSE_JSON>
{{
  "narrative": "Atmospheric prose here. No literal newlines.",
  "suggested_actions": [
    "Action 1",
    "Action 2",
    "Action 3"
  ],
  "state_updates": {{
    "character_name": "",
    "hp_change": 0,
    "sanity_change": 0,
    "mp_change": 0,
    "inventory_add": "",
    "inventory_remove": "",
    "location_name": "",
    "location_description": "",
    "clue_found": "",
    "clue_content": "",
    "thread_progress": ""
  }},
  "combat_update": {{
    "combat_started": false,
    "combat_ended": false,
    "attacker_name": "",
    "target_name": "",
    "attack_mode": "",
    "weapon_name": "",
    "weapon_damage": "",
    "skill_name": "",
    "defender_option": "",
    "shots_fired": 0,
    "bonus_dice": 0,
    "penalty_dice": 0
  }},
  "roll_request": {{
    "required": false,
    "skill_name": "",
    "action_text": "",
    "reason": ""
  }}
}}
</SYSTEM_RESPONSE_JSON>

Rules:
- No markdown.
- No commentary before or after the tags.
- JSON keys must remain in English.
- "narrative" and "suggested_actions" values must be in the selected session language.
- Integers only.
- Never use dice notation inside state_updates numeric values.
- weapon_damage may use dice notation like 1D3, 1D4, 1D6, 1D8, 1D10, 2D6+4, 4D6.
- If the narrative introduces a new distinct place that the investigators actually enter or clearly arrive in, fill location_name.
- If the narrative reveals a concrete, player-usable fact or evidence, fill clue_found and clue_content.
- If the narrative clearly advances an existing investigation line, fill thread_progress.
- Leave fields empty when nothing actually changed in state.

THREAD_PROGRESS FORMAT RULE
- thread_progress must be a short factual phrase describing what investigation line advanced.
- Do not express thread_progress as numeric fractions such as 1/4, 2/4, 3/4, etc.
- Do not use thread_progress as a scoreboard, tracker bar, or progress meter.
- Good examples:
  - "The anomalous isotope report is confirmed as genuine."
  - "The generator maintenance log becomes the new lead."
  - "The missing scientist is linked to the sealed sub-ice tunnel."
- Bad examples:
  - "Investigation progress 1/4"
  - "Anomaly thread: 2/4"
  - "Main mystery 50% complete"
  
STATE UPDATE ENFORCEMENT
- Use state_updates only for established changes, not for mere possibilities.
- Do not set clue_found merely because an object, journal, letter, diary, mirror, artifact, or lead is visible, named, suggested, or available to inspect.
- Set clue_found when the investigators actually learn, confirm, decode, or otherwise gain meaningful content from that source.
- Do not set location_name merely because a place is mentioned, proposed, glimpsed, or discussed.
- Set location_name when the investigators materially enter, reach, or clearly arrive at that place.
- If the response identifies the source of a lead, confirms a fact, or changes the active objective in a player-usable way, reflect that in state_updates.

COMBAT PAYLOAD RULES:
- Use combat_update only when violence, attack, armed threat, or an active combat exchange is occurring.
- attack_mode must be one of: "", "melee", "firearm", "maneuver", "other".
- attacker_name and target_name must name actors that already exist in CURRENT GAME STATE.
- Do not invent new enemies for combat_update. Use only actors from CURRENT GAME STATE.
- skill_name must be the exact existing skill name from the attacker’s sheet when possible.
- defender_option must be one of: "", "fight_back", "dodge", "dive_for_cover".
- Do not narrate final kill/destruction as automatic intent fulfillment. The backend resolves the mechanics.
- combat_started should become true when the first real attack or active hostile exchange begins.
- combat_ended should become true only when the fight is clearly over.

ROLL REQUEST RULES:
- Use roll_request when the player action is challenging and requires a roll before its outcome can be narrated.
- roll_request.required must be true only when a roll is actually needed now.
- skill_name must be the exact existing skill or attribute name from CURRENT GAME STATE.
- action_text should restate the gated action in a short plain form.
- reason should briefly explain why the roll is needed.
- When roll_request.required is true, do not narrate the successful completion of that action yet.
- If no roll is needed, set required=false and leave the other roll_request fields empty.
